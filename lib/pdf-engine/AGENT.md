# AGENT.md — PDF Engine

You are responsible for one isolated module: `lib/pdf-engine/`.
You do not need to know anything about the product that calls this module.
You will never read files outside this directory.

---

## What this module does

Generates PDF documents from typed data payloads.

- **No database access** — all data is passed in via the payload
- **No HTTP calls** except fetching photo URLs that are passed in the payload
- **No business logic** — caller decides what data to include, you render it faithfully
- **No imports from outside this directory** — ever

---

## Your only dependencies

- `pdf-lib` — PDF generation (already installed in the project)
- `node-fetch` or native `fetch` — for downloading photo URLs to embed
- Node.js built-ins only beyond that

Do not install or import anything else without confirming with the caller.

## Photo handling

When a `PDFPhoto[]` array is provided:

1. Fetch each URL with `fetch(url)` → `ArrayBuffer`
2. Try `pdfDoc.embedJpg()` first, fallback to `pdfDoc.embedPng()`
3. If fetch fails: render a grey placeholder box with caption "Photo unavailable"
4. Never throw — a failed photo must not abort the entire document

**Photo grid layout:**
- 1 photo: full width (483pt), max height 280pt, aspect-ratio preserved
- 2 photos: 2 columns, each 232pt wide, gap 8pt
- 3+ photos: 3 columns, each 151pt wide, gap 8pt, wrap to new row
- Caption below each photo: 8pt Helvetica #5C5C5C, centred, max 2 lines
- Attribution tag top-right corner of photo: small pill, "Landlord" (#2D6A4F bg) or "Tenant" (#16A34A bg), white 7pt text

## Template specifications

### 1. screening-report

**Purpose:** Full AI financial screening result for landlord (never shown to tenant).

**Structure:**
1. Cover page — "Financial Screening Report", property address, generated date, grade badge
2. Summary page — overall score dial/bar, grade, applicant names, rent vs income ratio
3. Per-applicant section — name, statement period, avg monthly income/outgoings, name verification status
4. Per-category breakdown — 6 sections (one per category), score bar, flags list, category summary
5. Verification page — QR code placeholder box + verification URL text, disclaimer

**Grade badge colours:**
- A: #16A34A (green)
- B: #65A30D (lime)
- C: #D97706 (amber)
- D: #EA580C (orange)
- F: #DC2626 (red)

**Score bar:** horizontal bar 300pt wide, filled proportionally, colour matches grade.

**Flags:** each flag renders as a row: severity icon (●) in colour + rule name bold + detail text. Severity colours: INFO=#9CA3AF, WARNING=#D97706, FAIL=#DC2626.

**Verification section:** grey bordered box, "Verify this report at:" label, full URL in monospace, note "This report is cryptographically linked to the original bank statement analysis."

---

### 2. inspection-report

**Purpose:** Property condition at tenancy start, dual-confirmed by landlord and tenant.

**Structure:**
1. Cover page — "Property Inspection", address, tenancy start date, AGREED/DISPUTED badge
2. Parties page — landlord block left, tenant block right, confirmation timestamps
3. Per-room pages — room name as section heading, condition badge, notes, photo grid (landlord photos first, then tenant photos, each attributed)
4. Confirmation page — "This report has been reviewed and confirmed by both parties" / dispute reason if DISPUTED, both signature blocks

**Condition badge colours:**
- GOOD: #16A34A background, white text
- FAIR: #D97706 background, white text
- POOR: #DC2626 background, white text

**DISPUTED status:** amber banner across cover "⚠️ DISPUTED — Tenant has not agreed to this report". All photos still included.

**Room page overflow:** if a room has many photos, continue on next page with room name repeated as continuation header.

---

### 3. apt-contract

**Purpose:** Assured Periodic Tenancy agreement (never AST — that term is abolished under RRA 2025).

**Legal notice:** Render at top of page 2, before clauses: "This agreement creates an Assured Periodic Tenancy under the Housing Act 1988 as amended by the Renters' Rights Act 2025. Fixed-term tenancies are no longer permitted for residential lets in England."

**Structure:**
1. Cover page — "Assured Periodic Tenancy Agreement", property address, tenancy start date
2. Parties page — landlord details left, all tenants listed right (up to 4), permitted occupiers if any
3. Key terms summary box — tenancy start, rent amount, rent due day, deposit amount, deposit scheme + reference
4. Clauses — numbered list, each clause: bold heading + body text. Page breaks inserted automatically when less than 80pt remains on page.
5. Special conditions (if any) — separate section with amber border
6. Signature page — one block per party: name, date line, signature line (or embedded image if signatureImageBase64 provided). Witness line below each.
7. Footer disclaimer on signature page: "This document was generated by LetSorted (letsorted.co.uk) from a pre-approved template. It should be reviewed by a qualified solicitor before signing if you have any doubts about its terms."

---

### 4. section-8-notice

**Purpose:** Statutory notice seeking possession — Housing Act 1988 s.8, as amended.

**Legal requirement:** This document's layout must match Form 3 (prescribed form) requirements. Render in a more formal/plain style — no cover page, start directly with the statutory header.

**Structure:**
1. Statutory header: "Notice Seeking Possession of a Property Let on an Assured Tenancy or an Assured Agricultural Occupancy" — centred, bold, 14pt. Reference: "Housing Act 1988 Section 8 as amended by the Housing Act 1996 and the Renters' Rights Act 2025"
2. Property and parties section
3. Grounds section — for each ground: ground number bold, statutory text of ground in smaller font (render the statutory wording — see below), then "Evidence:" label + landlord's evidence statement
4. Possession date — "Proceedings for possession will not begin before: [date]"
5. Signature block

**Statutory ground text to render (abbreviated — render these exactly):**
- GROUND_8: "Both at the date of the service of this notice and at the date of the hearing, at least two months' rent lawfully due from the tenant is unpaid."
- GROUND_10: "Some rent lawfully due from the tenant is unpaid on the date on which these proceedings are begun."
- GROUND_11: "Whether or not any rent is in arrears on the date proceedings are begun, the tenant has persistently delayed paying rent which has become lawfully due."
- GROUND_14: "The tenant or a person residing in or visiting the dwelling-house has been guilty of conduct causing or likely to cause a nuisance or annoyance to a person residing, visiting or otherwise engaging in a lawful activity in the locality."
- GROUND_7A: "The tenant or a person residing in or visiting the dwelling-house has been convicted of a serious offence, or has been found, by a court, to be in breach of an injunction to prevent nuisance and annoyance."
- GROUND_1: "The dwelling-house was occupied by the landlord or the landlord's spouse or civil partner as their only or principal home; or the landlord requires the dwelling-house as the only or principal home for the landlord or the landlord's spouse or civil partner."
- GROUND_2: "The dwelling-house is subject to a mortgage granted before the beginning of the tenancy and the mortgagee is entitled to exercise a power of sale conferred on the mortgagee by the mortgage or by section 101 of the Law of Property Act 1925."

**Style:** More austere than other templates. No brand green header. Plain border around document. Brand appears only in footer.

---

### 5. section-13-notice

**Purpose:** Statutory notice of rent increase — Housing Act 1988 s.13, as amended by RRA 2025.

**Legal requirement:** Must match Form 4 requirements.

**Structure:**
1. Statutory header: "Notice of Increase of Rent for a Periodic Assured Tenancy (other than a Statutory Periodic Tenancy)" — centred, bold
2. Reference: "Housing Act 1988 Section 13(2) as amended by the Renters' Rights Act 2025"
3. Property and parties
4. Rent increase details box: current rent, proposed rent, effective date, £ difference, % change
5. Last increase date if provided (to evidence 12-month gap)
6. Statement: "If you believe this increase is above the market rate, you may refer it to the First-tier Tribunal (Property Chamber) before the effective date."
7. Signature block

**Increase summary box:** bordered, background #F0F7F4. Two rows: Current and Proposed, with amounts in large bold text.

---

### 6. dispute-pack

**Purpose:** Multi-section evidence pack for deposit dispute, tribunal, or ombudsman.

**Addressed-to banner:** top of page 2, text varies:
- deposit_scheme: "Prepared for submission to [deposit scheme] Alternative Dispute Resolution service"
- tribunal: "Prepared for submission to the First-tier Tribunal (Property Chamber)"
- ombudsman: "Prepared for submission to the Property Ombudsman"
- court: "Prepared for use in County Court proceedings"

**Structure:**
1. Cover page — "Dispute Evidence Pack", property, parties, generated date, addressed-to label
2. Executive summary — dispute summary text, tenancy dates, deposit amount/scheme
3. Section A: Maintenance history — table of all tickets (date, category, priority, status, resolution time). Then per-ticket detail pages with photos for UNRESOLVED or URGENT tickets.
4. Section B: Rent payment history — table of all payments. Highlight LATE and PARTIAL rows in amber. Summary: total due, total paid, total outstanding.
5. Section C: Documents on file — table listing all documents with type, upload date, URL note.
6. Section D: Inspection vs Check-out comparison — if both photo sets provided, render side-by-side per room (inspection left column, check-out right column, same caption).
7. Section E: Event timeline — chronological list of all events, alternating row shading.
8. Section F: Template letters — two pre-drafted letters (ombudsman cover letter, tribunal statement template) with [FILL IN] placeholders. These are static text — render them regardless of addressed-to value.

**Section divider pages:** each section starts on a new page with a full-width brand green block, section letter large (60pt white), section title (20pt white).

## Testing

`lib/pdf-engine/test.ts` — generates one sample PDF per template with realistic mock data, writes to `/tmp/pdf-test/[template-name].pdf`.

Run with: `npx tsx lib/pdf-engine/test.ts`

All mock data should be realistic UK property data — real-looking names, addresses, amounts.
Every template must render without throwing. Photo URLs in test data should use a reliable public image (e.g. a Unsplash URL that won't 404).

## Error handling

- Never throw from `generatePDF` — return a `PDFResult` or throw a typed error with a `template` and `reason` field
- Failed photo fetches: render placeholder, continue
- Missing optional fields: render nothing for that section, continue
- Overflow: if content exceeds page, add new page automatically — never clip text
